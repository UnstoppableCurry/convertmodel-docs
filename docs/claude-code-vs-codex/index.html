Official Docs Comparison

Claude Code 和 Codex 到底差在哪

不要把它们都笼统理解成“会写代码的 CLI”。它们都能多步执行，但术语、产品边界和默认行为并不一样。

一张表讲清楚

Claude Code

  • 官方术语更常见的是 subagents、common workflows、git worktrees。
  • 更强调本地开发工作流、代理分工和工作树隔离。
  • 是否触发 subagent，和模型行为、客户端策略都有关系。

Codex

  • 更常见的是 profiles、provider、agentic coding、终端代理。
  • 用户更容易从模型/provider 配置角度理解它。
  • 即使请求兼容，也不代表能完全继承 Claude Code 的本地 agent 环境。

最容易混淆的地方

  1. 协议兼容 != 本地环境兼容。
  2. 模型会不会主动分派子代理，属于行为差异，不只是 API 差异。
  3. Claude Code 的多 agent 能力不是单纯靠服务端开关就能补出来。
  4. “task team” 不是两个产品统一的官方术语，更多是社区表达。阅读文档时优先看它们官方自己的词。

实际使用建议

  • 想要最稳：先把 Claude Code 或 Codex 的单线程路径跑通。
  • 想要兼容层：用统一网关做模型映射、鉴权和日志。
  • 想要多 agent：再检查本地 Git/worktree 条件，不要直接把责任推给网关。
